# Debug Session: image-scroll-jank

## Symptoms
- 纯文本模式滚动流畅。
- 图片模式滚动明显卡顿。
- 之前调整过滚动与快捷键路径后，图片模式仍然卡。

## Scope
- 仅聚焦图片模式滚动卡顿。
- 当前阶段禁止修改业务逻辑，只添加运行时埋点收集证据。

## Hypotheses
1. 图片卡顿主要发生在 `ClipCard` 重绘阶段，图片卡片在滚动时仍被频繁刷新。
2. 图片卡顿主要发生在缩略图解码阶段，`loadImageIfNeeded()` 的异步解码在滚动停止前后形成尖峰。
3. 图片卡顿主要发生在 SwiftUI 布局阶段，`LazyHStack` 在图片卡片存在时可见范围更新成本过高。
4. 图片卡顿主要发生在图片资源质量层，现有 `thumbnailData` 实际尺寸或格式不适合快速预览。
5. 图片卡顿主要发生在状态更新链路，滚动偏移、可见索引、预览开关之间的联动导致图片视图重复失效。

## Plan
1. 启动调试服务器并记录环境信息。
2. 为滚动、预览显示、图片解码、卡片重绘增加最小化埋点。
3. 请用户复现图片模式滚动卡顿。
4. 基于日志证据确认或排除上述假设。

## Evidence Summary
- 本次复现共产生 613 条日志，其中 A=335、B=52、C=18、D=3、E=205。
- 同一张图片卡片的 `image preview task fired` 被重复触发 15-19 次，但 `preview decode finished` 仅 1 次，说明主要问题是预览任务反复失效，不是每次都重新解码。
- `preview skipped while card hidden` 对同一张图片通常出现 8-9 次，说明滚动中“隐藏预览”的开关在持续抖动。
- `preview window moved` 只有 18 次，明显低于 335 次预览任务触发，说明真正的热点不在可见窗口移动本身，而在窗口内卡片反复切换预览状态。
- 大多数缩略图解码耗时在 1-5ms，但有 4 次解码同时出现在同一时间点且耗时约 565ms，说明停顿来自“解码尖峰”而非稳定持续的解码成本。
- 缩略图样本体积约 11KB-102KB，尺寸约 21x256 到 256x129，不足以单独解释滚动全程卡顿。

## Interim Conclusion
- 假设 A：确认。图片卡片在滚动期间被频繁重触发预览任务。
- 假设 B：部分确认。确有少量解码尖峰，但更像由任务成批恢复触发。
- 假设 C：基本排除为主因。可见窗口移动次数远小于预览任务次数。
- 假设 D：基本排除为主因。缩略图质量不是主要瓶颈。
- 假设 E：确认。`isActivelyScrolling` 与 `shouldShowPreview` 的联动导致预览反复失效。
